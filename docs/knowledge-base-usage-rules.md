# 知识库使用规范

> 本文档为第三层业务规则，智能体启动后必须完整阅读本文档才能开展后续操作。
> 后续规则迭代仅需修改本文档，无需重新下发启动提示词。

---

## 一、读取链路

智能体完成第二层知识库公约阅读后，必须按以下顺序读取第三层文档：

1. **知识库使用规范（本文档）** — 必须最先读，定义完整业务规则
2. **周总结**: GET /api/notes/系统使用规范/周总结.md
3. **月总结**: GET /api/notes/系统使用规范/月总结.md
4. **系统信息**: GET /api/notes/系统使用规范/系统信息.md
5. **各智能体更新总结**: GET /api/notes/智能体档案/[智能体名称]/更新总结.md

**Skill版本检查（自动执行）：**
- 读取系统信息后，检查本地skill版本号
- 如果skill版本低于知识库中记录的最新版本 → 自动更新skill
- 更新方式：读取知识库中的最新SKILL.md内容，覆盖本地skill文件

全部读取完成后，按下方汇报模板向用户提交结构化汇报。

## 二、汇报模板

智能体完成全部文档读取后，必须向用户提交以下结构化汇报：

```
📋 知识库连接汇报

**系统环境（公共信息）：**
- [从系统信息.md自动提取公共信息]

**我的密钥（归属信息）：**
- 智能体名称：[你的名称]
- API Key：[你的密钥]

**Hermes集群汇总摘要：**
- [周总结摘要]
- [月总结摘要]

**我的工作摘要：**
- [读取到的自己更新总结摘要]

**我的理解：**
- [简述你对任务的理解]

请确认以上信息是否准确，确认后我开始工作。
```

## 三、确认规则

- 用户回复任意肯定文字（如"确认""可以""开始""OK"等）= 确认通过
- 用户提出修改要求 → 修正汇报内容后重新提交确认
- 提交汇报后5分钟未收到回复 → 仅限有调度能力的智能体（如Hermes）提醒一次"汇报已提交，等待确认中"；无调度能力的智能体（如Codex、Claude Code）跳过提醒，等待用户主动回复

## 四、文档阅读规则

### 搜索与阅读
- 从用户任务指令中自动提取核心关键词搜索，不必每条都问用户确认
- 搜索接口: GET /api/search?q=关键词
- 搜索返回多个文档时，按创建时间由早至晚依次读取
- 全部文档阅读完毕后，整理核心要点并输出工作方案
- 等用户确认方案后再执行实际业务工作

### 读取优先级
- 公共文档（系统信息、周/月总结）优先于个人文档
- 相关性高的文档优先于低相关性文档

## 五、总结格式规范

### 更新总结（覆盖写入）
文件命名: `更新总结.md`

固定模板，四项全部必填：

```markdown
# 工作总结

## 最后更新: YYYY-MM-DD HH:MM

## 已完成工作
- [必填，至少一句话描述本次做了什么]

## 关键发现
- [必填，记录工作过程中的重要发现或结论]

## 待办事项
- [必填，列出后续需要跟进的事项，无则写"无"]
```

### 留痕总结（追加写入）
- 每次更新完成后，将更新总结的完整内容追加写入留痕总结

**追加操作步骤（必须执行）：**
1. GET 读取当前留痕总结内容
2. 在末尾追加新记录：`## YYYY-MM-DD HH:MM` + 更新总结全文
3. POST 完整内容覆盖写入

- 如果 GET 返回 404（首次写入），直接 POST 新内容
- 日常初始化与业务运行过程均不加载留痕总结

### 双总结同步约束
- 更新总结内容必须与留痕总结内最新一条记录完全一致
- 更新总结仅存储最后一版，降低算力消耗
- 更新总结承担原"开发记录"的留存功能，不再单独设立开发记录文档

**写入后回读验证（必须执行）：**
1. POST 更新总结后，立即 GET 回读，对比内容是否一致
2. POST 留痕总结后，立即 GET 回读，确认追加成功
3. 回读不一致 → 重新POST，最多重试2次
4. 两次重试仍失败 → 在汇报中注明"总结写入验证失败"，继续其他工作

## 六、指令触发规则

| 用户指令 | 智能体行为 |
|----------|-----------|
| "更新总结" | 立即更新自己的两套总结文档（更新总结+留痕总结） |
| "结束工作" | 复查本次是否已更新总结和系统信息，未更新则强制完成后终止工作 |
| （无特殊指令） | 全部业务工作结束后自动执行总结更新操作 |

## 六-3、Skill安装与自动加载

新智能体安装知识库 Skill 后，必须配置自动加载，否则后续会话不会触发连接流程。

**安装步骤：**
1. GET /api/notes/系统使用规范/skill-当前版本.md — 获取最新 skill 全文
2. 将内容保存为本地 skill 文件
3. 配置自动加载：
   - Hermes Agent：`hermes config set skills.preload rtyu-knowledge-base`
   - Codex：将 skill 保存到 `~/.codex/skills/rtyu-knowledge-base/`
   - 其他环境：按各自方式配置 skill 自动加载
4. 重启会话验证 skill 是否自动触发

**版本检查：**
- 读系统信息中 `Skill版本` 字段（YYYYMMDDNN 格式）
- 对比本地 skill 的 `version` 字段
- 本地版本 < 知识库版本 → 尝试自动更新
- 更新失败（权限被拒）→ 在汇报中注明"Skill版本过旧，请管理员更新"

## 七、收尾更新规范

### POST请求统一规范
- Content-Type: application/json
- Header: Authorization: Bearer [你的API Key]
- 失败处理：按异常处理分类策略执行，不统一重试

### 三类POST写入操作

**第一类：更新总结（必填）**
```
POST /api/notes/智能体档案/[智能体名称]/更新总结.md
Content: {"content": "# 工作总结\n\n## 最后更新: YYYY-MM-DD HH:MM\n\n## 已完成工作\n- ...\n\n## 关键发现\n- ...\n\n## 待办事项\n- ..."}
```

**第二类：留痕总结（必填）**
```
POST /api/notes/智能体档案/[智能体名称]/留痕总结.md
Content: {"content": "# 工作留痕总结\n\n[追加本次更新总结完整内容]"}
```

**第三类：系统信息（结束工作必填）**
```
POST /api/notes/系统使用规范/系统信息.md
Content: {"content": "# 系统信息\n\n[覆盖更新]"}
```

### "结束工作"复查判定 / "Finish Work" Review

收到"结束工作"指令时，执行收尾五步：
When receiving "finish work" command, execute 5 closing steps:

1. **更新总结**（覆盖写入）/ Update Summary (overwrite)
2. **留痕总结**（追加写入）/ Trace Summary (append)
3. **系统信息**（覆盖写入）/ System Info (overwrite)
4. **会客厅签到**（发消息到会客厅）/ Lounge Check-in (post message)
5. **图谱标签**（POST /api/graph/tags）/ Graph Tags (set tags)

已更新 → 跳过步骤1-3，只做步骤4-5
Already updated → Skip steps 1-3, only do 4-5

未更新 → 全部执行
Not updated → Execute all 5 steps

## 八、通用约束

### 异常处理

**错误分类与策略：**

| 错误类型 | HTTP 状态码 | 处理策略 |
|----------|------------|----------|
| 认证失败 | 401 | 立即终止，上报"API Key 无效"，不重试 |
| 权限不足 | 403 | 立即终止，上报具体权限缺失，不重试 |
| 资源不存在 | 404 | 按业务逻辑处理（如新智能体创建档案），不重试 |
| 服务器繁忙 | 502/503/504 | 指数退避重试：5s → 10s → 20s，最多3次；仍失败上报"服务器过载，建议人工检查" |
| 超时 | 无响应 | 指数退避重试：5s → 10s → 20s，最多3次；仍失败上报"接口超时：[具体URL]" |
| 其他 5xx | 500/5xx | 重试3次间隔5秒，仍失败上报具体错误 |
| 空内容 | 200 但无数据 | 重试1次，仍为空则上报"返回空内容：[具体URL]" |

**通用规则：**
- 所有失败最终需向用户报告具体错误信息（哪个接口、什么错误）
- 报告格式：`[接口名称] 失败：[HTTP状态码] [错误摘要]`
- 避免静默失败，暂存的本地数据不丢弃，待用户指示后续操作

### 智能体名称

**身份绑定规则：**
- API Key 是智能体的全局唯一身份标识，100% 绑定
- 名称由 Key 反查确定：智能体启动时调用 GET /api/agent/profile，系统根据 Key 返回对应名称
- 智能体不得自行声明或修改自己的名称，一切以 API Key 反查结果为准

**反查机制：**
- 密钥管理系统中每把 Key 标记归属名称 → API Key 即身份
- 名称仅为前台显示标识，方便用户识别；后端逻辑以 Key 为准
- Key 与名称不一致时（如前台改名未换 Key），以 Key 登记的原始名称为准

**路径使用：**
- 名称用于文件路径（如 `智能体档案/[名称]/`）
- 名称中的特殊字符可能导致路径错误，创建时避免使用特殊字符

### 并发写冲突处理

**公共文档写入保护：**
- 写入前先 GET 读取最新版本，记录已知内容
- 写入失败（503/连接中断）：重试前再次 GET 确认是否已被其他智能体更新
- 如果发现内容已变更：将待写入内容与最新版本合并，或保留自己的工作内容等待用户指示
- 乐观锁建议：服务端支持时使用 If-Match 或版本号检测，避免盲覆盖

### 流程中断恢复
- 不需要特殊机制，智能体中断后重新执行完整的三层读取链路
- 汇报时用户可以告知上次做到哪了，从断点继续

### 权限规则
- 公共文档（系统信息、周/月总结）：所有智能体可读可更新
- 用户留言：仅定时任务（Hermes定时任务）读取处理，智能体启动时不读取
- 他人档案：仅Hermes可读，其他智能体不可读
- 自己档案：各自读写
- 知识库使用规范修改：智能体可提出修改建议，需用户确认后才能改

### 文档命名规则
- 更新总结: `更新总结.md`
- 留痕总结: `留痕总结.md`（固定名）
- 周总结: `系统使用规范/周总结.md`（固定名）
- 月总结: `系统使用规范/月总结.md`（固定名）

## 九、安全事故教训

71. **轻量轮询必须取消**——codex跑轻量轮询时，因只读了部分文档，差点把用户开发的其他系统全覆盖。所有智能体必须走全量读取链路，不允许轻量轮询。
72. **/api/docs 不能硬编码管理员key**——任何人都能访问 /api/docs 获取管理员key，然后读取系统信息中的全量密码、Token。api_key字段改为从连接提示词获取。
73. **连接提示词步骤数必须与/api/docs对齐**——连接提示词说4步，/api/docs说14步，智能体搞不清"第一步"指哪个。已明确"连接提示词Step1 → /api/docs next_steps 14步 → 汇报确认"。
