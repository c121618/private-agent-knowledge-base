# 目录维护规范

> 本文档定义知识库分类目录的结构、维护策略和生命周期管理。
> 各智能体独立轮询，前台可配置周期参数。
> 采用"自维护"模式：每个智能体在空闲时自行重建自己常用分类的目录。

---

## 一、目录结构

每个分类目录下的 `目录.md` 分三区：

```markdown
# 技术方案目录

> 自动生成于: 2026-06-22 12:00
> 方案总数: 12

## 🆕 新方案（3个月内）
| 方案 | 关键词 | 调用记录 | 状态 | 路径 |
|------|--------|----------|------|------|
| API限流 | rate-limit, throttle | 2026-06-20 ✅ | ✅已验证 | 归档/技术方案/API限流.md |
| 缓存策略 | redis, cache | 无 | ⚠️未验证 | 归档/技术方案/缓存策略.md |

## ✅ 已验证方案（>3个月，有成功记录）
| 方案 | 关键词 | 最近调用 | 总计调用 | 路径 |
|------|--------|----------|----------|------|
| 日志收集 | elk, logstash | 2026-04-15 | 8次 | 归档/技术方案/日志收集.md |

## 📦 归档方案（>3个月，无调用/不通过）
| 方案 | 原因 | 路径 |
|------|------|------|
| 旧版限流 | 被API限流替代 | 归档/技术方案/旧版限流.md |
| 手动备份 | 验证不通过，已弃用 | 归档/技术方案/手动备份.md |
```

### 三区界定标准

| 分区 | 时间条件 | 调用条件 |
|------|----------|----------|
| 🆕 新方案 | 创建 < 3个月 | 任意 |
| ✅ 已验证 | 创建 ≥ 3个月 | 有成功调用记录 |
| 📦 归档 | 创建 ≥ 3个月 | 无调用 或 调用全失败 或 已有替代方案 |

### 状态标记

| 状态 | 含义 | 所在分区 |
|------|------|----------|
| ✅ 已验证 | 有成功调用记录 | 新方案区或已验证区 |
| ⚠️ 未验证 | 创建后无人使用 | 新方案区 |
| ❌ 不通过 | 调用全部失败 | 归档区 |
| 🔄 已替代 | 被新方案取代 | 归档区 |

## 二、元数据头规范（每篇方案文档必填）

每篇方案文档开头必须有标准元数据块：

```markdown
> 分类: 技术方案
> 关键词: rate-limit, api, throttle
> 创建: 2026-06-22
> 用途: 解决API接口被频繁调用时的限流问题
```

这样的好处：定时任务扫描时不需要全文分析，读前3行就提取完毕。

### 字段说明

#### 多分类说明

一个文档可以属于多个分类。元数据头中写逗号分隔：

```markdown
> 分类: 运维, 技术方案
> 关键词: postgresql, timeout, 连接池
> 创建: 2026-06-22
> 用途: 解决PG连接超时问题
```

效果：
- 该文档会出现在 `归档/运维/目录.md` 和 `归档/技术方案/目录.md` 两个目录中
- 无论用户从"运维"还是"技术方案"分类检索，都能搜到这篇方案
- 不需要全局搜索，不需要跨分类遍历

#### 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| 分类 | 是 | 与目录路径一致，支持多分类（逗号分隔，如"运维, 技术方案"） |
| 关键词 | 是 | 逗号分隔，供搜索匹配 |
| 创建 | 是 | 文档创建日期，用于判断3个月界限 |
| 用途 | 否 | 一句话说明解决什么问题 |

## 三、调度机制

### 3.1 轮询策略

各智能体在业务空闲时自行执行目录轮询。

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 轮询间隔 | 每6小时 | 可前台配置：1小时 / 6小时 / 每天 |
| 轮询范围 | 按分类 | 自己常用分类优先，全库可选 |
| 并行度 | 每次一个分类 | 避免多个智能体同时写同一目录 |

### 3.2 轮询方法

```
轮询触发（定时到达 / 用户手动触发"更新目录"）
        │
        ▼
① 读取目标分类的所有文档
   GET /api/search?q=分类:归档/<分类名>
   （按路径前缀过滤，仅取该分类下的文档）
        │
        ▼
② 逐篇检查
   ├── 提取元数据头（分类、关键词、创建日期）
   └── 提取文档尾部的调用记录表
        │
        ▼
③ 分类到三区
   ├── 创建 < 3个月 → 🆕 新方案区
   ├── 创建 ≥ 3个月 + 有成功调用 → ✅ 已验证区
   └── 创建 ≥ 3个月 + 无调用/失败 → 📦 归档区
        │
        ▼
④ 重建目录.md
   POST /api/notes/归档/<分类名>/目录.md
```

### 3.3 配置参数（前台可调）

修改本文档即可全局生效：

```
轮询间隔: 6小时
新老界限: 3个月
轮询范围: 本智能体常用分类
```

各智能体读取本文档获取当前配置。

## 四、生命周期的流转

### 4.1 方案从新到老的完整路径

```
🏷 创建新方案
   存入 归档/<分类>/<标题>.md
   （创建时间写入元数据头）
        │
        ▼
   🆕 新方案区（0~3个月）
        │
        ├── 被调用 → 追加调用记录 → 保持新方案区
        │                   │
        │                   ▼
        │              3个月后 → 移到 ✅ 已验证区
        │
        └── 无人调用 → 3个月后 → 移到 📦 归档区
```

### 4.2 调用对状态的影响

| 场景 | 效果 | 说明 |
|------|------|------|
| 新方案有首次成功调用 | 保持🆕区，标记✅已验证 | 等待3个月再升到已验证区 |
| 归档方案突然被调用成功 | 从📦区升回🆕区 | 重新计算3个月窗口 |
| 已验证方案长期无人调用 > 3个月 | 从✅区降到📦区 | 降级为归档方案 |

### 4.3 未解决/失败的记录

调用结果为 ❌失败的方案：

```
📦 归档区 → 标记❌不通过
             ↓
下次搜到同问题时 → 输出时附带说明：
"此前有方案[XXX]尝试解决但失败（2026-03）：
  原因：方案不适配当前环境
  避免：不要重复此路径，尝试以下替代方案..."
```

失败本身就是有价值的信息——告诉后来者这条路走不通，不用再试。

## 五、多智能体协作规则

### 5.1 写冲突保护

目录写入前先 GET 最新版本，再 POST 覆盖：
```
GET 归档/<分类>/目录.md
→ 读取当前内容
→ 合并自己的新扫描结果
→ POST 覆盖写入
```

如果 POST 时发现已被其他智能体更新：
→ 重新 GET 最新版
→ 重新合并
→ 再次 POST

### 5.2 轮询频率错开

- 每个智能体轮询时随机等待 5~30 秒再写入
- 避免两台同时扫同一个分类
