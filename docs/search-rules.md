# 检索规范

> 本文档定义智能体处理业务问题时的检索优先级、约束条件和分级策略。
> 修改本文档即可全局调整检索行为，无需改动任何智能体提示词。
> 所有智能体在处理业务问题前，必须先查阅本文档确认当前检索策略。

---

## 一、检索总则

### 1.1 检索优先级

```
用户提出业务问题
       │
       ▼
┌─────────────────────────────────────────────┐
│ Tier 0: 精确匹配（目录关键词列）             │
│   → 命中 → 读方案输出                       │
│   → 不中 → 进入 Tier 1                      │
└─────────────────────────────────────────────┘
       │（不中）
       ▼
┌─────────────────────────────────────────────┐
│ Tier 1: 模糊匹配词典扩展（通用，零算力）     │
│   读 检索配置/模糊匹配词典.md               │
│   → 词典有匹配 → 提取扩展关键词             │
│           → 回目录做精确匹配                 │
│           → 命中 → 读方案输出               │
│           → 不中 → 进入 Tier 2              │
│   → 词典无匹配 → 直接进入 Tier 2            │
└─────────────────────────────────────────────┘
       │（不中）
       ▼
┌─────────────────────────────────────────────┐
│ Tier 2: 跨分类扫描（多读几个目录表）         │
│   本分类不中 → 依次读相邻分类目录            │
│   → 任一命中 → 读方案输出                   │
│   → 全都不中 → 进入 Tier 3                  │
└─────────────────────────────────────────────┘
       │（不中）
       ▼
┌─────────────────────────────────────────────┐
│ Tier 3: 语义理解（仅能力达标的智能体执行）   │
│   LLM用自己的理解力判断目录内容与问题的关联   │
│   → 命中 → 输出方案                         │
│          → 把本次"用户原话→命中关键词"       │
│            追加到模糊匹配词典 💡              │
│   → 不中 → 联网搜索（进入第三章）           │
│                                              │
│   能力不达标的智能体 → 直接跳到联网搜索      │
└─────────────────────────────────────────────┘
       │（不中）
       ▼
┌─────────────────────────────────────────────┐
│ ④ 联网 → 归档入库 → 更新目录                │
└─────────────────────────────────────────────┘
```

### 1.2 核心原则

- **知识库是默认第一信息源**，外网仅作为空白场景的补充
- **目录导航优先**：按分类读目录 → 匹配关键词 → 定位文档，不搜全文、不问用户
- **模糊匹配词典兜底**：Tier 1 用词典（一串字符包含判断）扩展用户口语到关键词，零算力
- **进化反哺**：Tier 3 语义解决后必须反哺词典，让简单智能体下次也能直接命中
- **交叉校验**：同一分类内多份方案互相佐证（非强制，有则用）
- **库无即联网**：所有层级均无匹配 → 联网搜索。找到后归档入库、更新目录
- **调用留痕**：任何方案被使用后必须记录结果，形成实战信誉分

## 二、知识库检索（第一优先级）

### 2.1 触发条件

收到业务问题后，**必须**先执行知识库检索，不得跳过直接联网。

### 2.2 检索流程

> 不用搜索API，用目录导航。目录信息必须完整，Agent只查目录表，不扒文件。
> 目录只查更新时间——没变不重读，变了才刷新。

**Tier 0：精确匹配（所有智能体）**

```
① 判定问题所属分类
   运维 / 技术方案 / 业务规则 / 其他
   GET /api/notes/归档/<分类>/目录.md

② 在目录关键词列精确匹配
   → 有匹配 → 进入④ 读方案
   → 无匹配 → 进入 Tier 1
```

**Tier 1：模糊匹配词典（所有智能体，零算力）**

```
① GET /api/notes/检索配置/模糊匹配词典.md

② 检查用户问题是否包含词典`用户说`列中的词
   → 包含 → 提取对应`扩展关键词`
          → 用扩展关键词回目录做精确匹配
          → 命中 → 进入④ 读方案
          → 不中 → 进入 Tier 2
   → 不包含 → 直接进入 Tier 2
```

> 词典匹配就是简单的字符串包含判断，任何智能体都能执行。
> 不需要分词、不需要语义、不需要向量，就是一个文件的内存查找。

**Tier 2：跨分类扫描（所有智能体）**

```
① 依次读取相邻分类的目录.md
   ├── 不搜"归档/其他/"（兜底分类）
   └── 读取优先级：根据问题语义倾向
       → 运维问题 → 先读运维目录
       → 技术问题 → 先读技术方案目录

② 每个目录表做精确匹配（关键词列）
   → 任一命中 → 进入④ 读方案
   → 全都不中 → 进入 Tier 3
```

> 目录表很小（一个分类通常几十行），读3~5个目录就是几百字节。
> 成本：N次GET请求，零计算开销。

**Tier 3：语义理解（能力达标的智能体执行）**

```
① 再次读取各分类目录表

② 用语义理解判断目录关键词列与用户问题的关联
   → 感知到关联 → 命中 → 进入④ 读方案
          └ 输出后执行进化反哺（见 2.6）
   → 无关联 → 进入第三章 联网搜索

③ 能力不达标的智能体：
   跳过Tier 3，直接进入第三章 联网搜索
```

> 语义理解是**可选层**。能做的智能体做了后必须反哺词典；不能做的跳过即可，不影响结果。

**④ 读方案（时间窗口）**

```
从匹配到的方案中，按目录时间倒序读最近10篇
不足10篇 → 扩展到1个月内创建/更新的方案
阅读方案文档时：
▶ 目录表已有调用记录摘要：最近调用时间 + 信誉分
→ 信任分高 → 直接读原文方案输出
→ 久未调用/未验证 → 标注⚠️

仍不能解决 → 联网搜索（进入第三章）
```

> 调用记录摘要已包含在目录表中（最近调用时间+信誉分），Agent读目录即可判断方案可信度。
> 不需要打开文档查调用记录——省算力。

### 2.3 库内解决判定标准

满足以下任一条件即判定为"库内可解决"：

| 条件 | 说明 |
|------|------|
| 完全匹配 | 知识库有完全相同问题的解决方案 |
| 部分匹配+可组合 | 多份文档组合后可拼出完整方案 |
| 相似匹配+微调 | 有相似案例，修改参数后可适用 |
| 经验佐证 | 最近3份总结对该问题的处理结论一致 |

### 2.4 方案输出（双流结构）

检索到方案后，输出分为**两股信息流**：

```
结果输出格式：

📋 [问题] 的检索结果

── 方案流（最新10份总结） ──
① [方案标题]（YYYY-MM-DD）— 来源：[文档路径]
   摘要：[核心步骤/结论]
   📊 使用记录：N条（成功X / 失败Y）← 实战信誉分

② [方案标题]（YYYY-MM-DD）
   ...

── 使用记录流（调用证据） ──
• [YYYY-MM-DD] 智能体 [名称] 调用此方案 → ✅ 成功
  [简要描述使用场景和结果]
• [YYYY-MM-DD] 智能体 [名称] 调用此方案 → ⚠️ 部分成功
  [需要手动调整的部分]
• [YYYY-MM-DD] 智能体 [名称] 调用此方案 → ❌ 失败
  [失败原因和最终采用的替代方案]
```

- 方案流：时间倒序展示最新方案
- 使用记录流：展示每条方案的实战结果，按时间倒序
- 无使用记录时标注"⚠️ 此方案尚未有调用记录"

### 2.5 使用记录文件格式

调用了知识库方案后，必须记录使用结果：

```
POST /api/notes/知识库/使用记录/<YYYY-MM-DD>-<问题关键词>.md

内容模板：
# [问题] — 使用记录

**使用时间**: YYYY-MM-DD HH:MM
**使用智能体**: [名称]
**来源方案**: [原始方案的文档路径]

## 调用结果
- 状态: ✅ 成功 / ⚠️ 部分成功 / ❌ 失败
- 详细描述: [实际使用情况，包括遇到的问题和如何解决的]

## 方案验证
- 方案完整度: 完全按步骤执行 / 需要调整部分步骤
- 执行耗时: [实际操作耗时]

## 改进建议
- [方案有哪些可优化的地方]
```

### 2.6 进化反哺机制

> 这是知识库自我进化的核心闭环。
> 词典越用越准、越用越全，直至覆盖所有常见模糊表达。

**触发条件**：智能体在 Tier 3（语义理解）中命中方案并成功解决问题后，必须执行反哺。

**执行步骤**：

```
① 解决问题后，记录：
   - 用户原话（当时怎么问的）
   - 命中的关键词（目录中哪一列匹配上的）

② 追加到模糊匹配词典：
   GET /api/notes/检索配置/模糊匹配词典.md
   → 读取当前全文
   → 在末尾追加：
     | 用户说（模糊）       | 扩展关键词（精确）       | 录入人 |
     |----------------------|-------------------------|--------|
     | 用户的原话关键词     | 本次命中的关键词         | [名称] |
   → POST 覆盖写入

③ 更新自己的更新总结
```

**约束**：
- 同一用户原话已被词典覆盖 → 不重复追加
- 追加前检查词典是否已有该词
- 能力不达标的智能体不需要执行；用词典解决即可

**进化效果**：
```
首次：用户说"连不上" → 词典无匹配 → Tier 3语义理解 → 解决
      → 反哺："连不上" 追加到词典
第二次：用户说"连不上" → 词典有匹配 → 精确命中 → 解决 ✓
第三次（其他智能体）：用户说"连不上" → 词典有匹配 → 精确命中 → 解决 ✓
```

### 2.7 后处理约束：方案文件内标记总结

> 方案被使用并解决问题后，必须回到**原始方案文档本身**留下调用记录。
> 让每条方案自带"实战记录"，后续搜索时一打开就能看到所有历史使用情况。

#### 执行步骤

```
① 使用知识库方案 → 解决问题
② 回到原始方案文档：GET 原始方案文档路径
③ 在文档底部追加本条调用记录
   → 追加格式见下方模板
④ POST 覆盖写入更新后的文档
⑤ 同步更新自己的更新总结
```

#### 调用记录追加格式

在原始方案文档末尾追加以下内容：

```markdown
---

## 调用记录

| 时间 | 智能体 | 结果 | 场景 | 备注 |
|------|--------|------|------|------|
| YYYY-MM-DD HH:MM | [名称] | ✅ 成功 | [问题场景] | [备注] |
```

结果标记：
| 标记 | 含义 |
|------|------|
| ✅ 成功 | 方案完全适用，直接解决 |
| ⚠️ 部分成功 | 主体方案可用，需微调部分参数/步骤 |
| ❌ 失败 | 方案不适用于当前场景 |

#### 示例

```markdown
# PostgreSQL 连接超时解决方案

## 问题描述
...

## 解决步骤
1. ...

---

## 调用记录

| 时间 | 智能体 | 结果 | 场景 | 备注 |
|------|--------|------|------|------|
| 2026-06-22 10:30 | codex | ✅ 成功 | 知识库CT202连接超时 | 完全按步骤执行，直接解决 |
| 2026-06-21 15:20 | code | ✅ 成功 | VM100数据库连接问题 | 步骤2需要将timeout改为60s |
| 2026-06-20 09:00 | Hermes | ⚠️ 部分成功 | Gitea服务连接超时 | 步骤3不适合容器环境，改用pct exec |
```

#### 约束规则

- **每次使用方案后必须追加**，不得遗忘
- 调用记录只追加、不删除，保留完整使用历史
- 如果方案被更新（内容本身修改），调用记录**保留不动**，只能追加新行
- 从未被使用的方案无调用记录表格

## 三、外网检索（自动兜底）

### 3.1 触发条件

> 所有 Tier 检索完成后无匹配 → **自动跳外网**。
> 不需要用户授权。

满足**任一**条件即启动：
1. 目录精确匹配 + 词典扩展 + 跨分类扫描（+ 语义理解）全部无结果
2. 读方案（时间窗口）后仍无法解决

### 3.2 检索后处理

```
① 外网找到有效信息
   → 逐条核验网页内容有效性
   → 输出标准化完整方案

② 执行强制归档（见第四章）
   → 归档到对应分类
   → 写入元数据头（来源标注为"外网检索"）
   → 设置图谱标签

③ 更新目录
   → 方案已入库 → 下次目录轮询自动纳入目录表
   → 并附带调用记录（时间+结果）

④ 通知用户：
   "已从网络检索并归档到知识库，
    后续本分类目录即可检索到此方案。"
```

## 四、强制归档

### 4.1 归档触发条件

满足**任一**即执行：
1. 外网检索完成 → 结果必须归档
2. 用户说"保存"/"记录一下" → 立即归档
3. Hermes 定时检查 → 批量归档

### 4.2 归档时关键词扩展（新）

> 归档文档时，连带做关键词扩展。
> 扩展发生在归档时（��次），查询时（无数次）直接精确命中。

```
① 撰写方案后
② 检查模糊匹配词典中当下分类的常用用户口语
③ 将这些口语词汇写入文档元数据头的`关键词`列
④ 效果：
   用户说"数据库卡死了" → 词典扩展 → 精确匹配→ 命中
                        ← 关键词列中有"卡死"
```

**扩展原则**：
- 不依赖词典也可人工扩展（写入者根据自己的经验补充）
- 扩展的关键词写在元数据头的`关键词`列，逗号分隔
- 不要写得太多（最多5~8个扩展词），避免误匹配

### 4.3 归档格式

归档文档使用统一模板：

```markdown
# [问题标题]

**归档时间**: YYYY-MM-DD HH:MM
**归档智能体**: [名称]
**信息来源**: 知识库经验 / 外网检索 / 用户提供

## 问题描述
[问题的完整描述]

## 解决方案
[标准化的解决步骤/方案]

## 关键结论
[提取的核心结论和注意事项]

## 标签
[标签1, 标签2, ...]
```

### 4.4 归档位置

```
归档/
├── 技术方案/     ← 技术类归档
├── 业务规则/     ← 业务逻辑归档
├── 运维记录/     ← 运维经验归档
└── 其他/         ← 未分类

使用记录/
├── 技术方案/     ← 技术方案的调用结果
├── 业务规则/     ← 业务规则的调用结果
└── 运维记录/     ← 运维操作的调用结果
```

### 4.5 归档后操作

1. POST 写入归档文档
2. POST 设置图谱标签（含问题分类标签，用于与使用记录关联）
3. 如已有使用记录→读取并追加到方案输出中
4. 更新自己的更新总结和留痕总结

## 五、配置参数（可前端调整）

以下参数修改本文档即可全局生效：

| 参数 | 当前值 | 说明 |
|------|--------|------|
| 知识库导航方式 | 目录匹配 + 词典扩展 | 先精确匹配，再词典扩展，再跨分类，再语义 |
| 模糊匹配词典 | 检索配置/模糊匹配词典.md | 全局共享，进化式增长 |
| 词典匹配方式 | 字符串包含（零算力） | 任何智能体都能执行 |
| 跨分类扫描范围 | 本分类 + 相邻分类 | 可配置扫描哪些分类 |
| 语义理解 | 可选（能力达标的执行） | 不达标者跳过直接联网 |
| 进化反哺 | 必须执行（能力达标者） | 语义解决后反哺词典 |
| 归档时关键词扩展 | 建议执行 | 归档时顺手扩展，减少未来模糊查询 |
| 交叉校验 | 可选 | 同分类多方案互相佐证，不强制 |
| 外网兜底年龄阈值 | 30天 | 超过此天数不考虑知识库方案 |
| 归档分类前缀 | 归档/ | 归档文档存储根目录 |
| 使用记录目录 | 使用记录/ | 使用记录文档存储根目录 |
| 强制归档必须执行 | true | 外网检索后是否必须归档 |
| **调用留痕必须执行** | **true** | **使用知识库方案后必须记录结果** |
| **双流输出必须执行** | **true** | **搜索结果必须同时输出方案流+使用记录流** |

## 六、注意事项

- 检索应串联到触发条件中（见 智能体连接提示词 + 连接状态判定）
- 归档文档需要定期整理，避免过度碎片化
- 同一问题的多个归档版本，以最新一份为准
- **使用记录与原始方案必须共享同一"问题分类"标签**，否则图谱无法自动关联
- 使用记录的"失败"状态同样重要——它能帮助其他智能体避开同样的坑
- **进化反哺是长期竞争力的关键**：词典越全，简单智能体也能处理的场景越多
